# Git 提交代码

## ⚠️ 严禁事项

**绝对禁止 1**：在执行 git 提交流程时，**严禁修改任何代码文件**。只能执行 git 操作（pull、add、commit、push）和分析代码变更，绝对不能修改、编辑、格式化或重构任何代码文件。如果发现代码问题，只能提示用户，不能自行修改。

**绝对禁止 2**：**严禁跳过步骤 5（用户确认步骤）直接执行 commit 和 push**。这是最危险的违规行为，可能导致错误的代码被提交到远程仓库。**必须**在步骤 5 中展示 commit message 并等待用户明确确认后，才能执行步骤 6（commit）和步骤 7（push）。如果 AskQuestion 工具不可用，必须通过明确询问用户的方式获取确认，绝对不能跳过或省略这个步骤。

## 提交流程

执行以下完整流程来提交代码：

### 步骤 0: 代码审核

**重要**：在执行 git pull 之前，必须先进行代码审核。

1. **查看待提交的代码变更**：
   ```bash
   git status
   git diff
   ```
   查看所有未提交的代码变更

2. **读取代码审核规则**：
   - 读取并应用代码审核规则文件：`.cursor/skills/git-safe-commit/code-review-rules.mdc`
   - 根据规则文件中的审核要点、审核标准和审核流程进行代码审核

3. **严禁修改代码**：只能识别和报告问题，绝对不能修改任何代码文件

4. **审核结果处理**：
   - **如果审核通过**：
     - 继续执行步骤 1（git pull）
   
   - **如果审核发现问题**：
     - 列出所有发现的问题
     - 明确告知用户："代码审核发现问题，请先修复后再提交"
     - 提供具体的问题描述和建议
     - **严禁修改代码**：绝对不能自行修改代码，只能提示用户修复
     - **不要继续执行后续步骤**，等待用户修复问题
   
   - **如果审核规则文件不存在或为空**：
     - 提示用户："代码审核规则文件尚未配置，跳过代码审核"

**重要**：代码审核是提交流程的第一步，必须在 git pull 之前完成。只有审核通过或规则未配置时，才继续后续步骤。

### 步骤 1: 拉取远程分支
```bash
git pull
```
确保本地代码与远程同步，避免冲突。

### 步骤 2: 检查代码冲突

**重要**：在继续之前，必须检查是否有代码冲突。需要区分真正的冲突标记和格式问题（如 trailing whitespace）。

1. **检查 git 状态**：
   ```bash
   git status
   ```
   查看是否有冲突文件（显示为 "both modified"、"both added"、"Unmerged paths" 等状态）

2. **检查冲突标记**：
   ```bash
   git grep -n "<<<<<<< HEAD" || git grep -n "=======" || git grep -n ">>>>>>>"
   ```
   如果找到冲突标记（`<<<<<<<`、`=======`、`>>>>>>>`），说明有未解决的冲突

3. **判断是否有冲突**：
   - **有冲突**：`git status` 显示 "Unmerged paths" 或包含 "both modified"、"both added" 等状态，或者找到冲突标记
   - **无冲突**：继续下一步（注意：trailing whitespace 等格式问题不是冲突，不影响提交）

5. **处理冲突**：
   - **如果有冲突**：
     - 立即停止提交流程
     - 明确告知用户："检测到代码冲突，请先解决冲突后再提交"
     - 列出所有冲突文件
     - 提示用户可以使用 `git status` 查看冲突文件
     - **不要继续执行后续步骤**，等待用户解决冲突
   
   - **如果没有冲突**：
     - 继续执行步骤 3

### 步骤 3: 暂存所有更改
```bash
git add .
```
将所有修改的文件添加到暂存区。

### 步骤 4: 分析代码变更并生成 commit message

**重要**：在执行 `git commit` 之前，必须先分析代码变更来生成合适的 commit message。

1. **查看代码变更**：
   ```bash
   git diff --cached
   ```
   或者查看未暂存的变更：
   ```bash
   git diff
   ```

2. **分析变更内容**：
   - 查看修改的文件列表和变更类型
   - 识别主要功能变更、bug 修复、重构等
   - 确定合适的 commit type

3. **获取当前分支名并提取数字**：
   ```bash
   git rev-parse --abbrev-ref HEAD
   ```
   从分支名中提取所有连续数字（例如：`feature/17245-something` → `17245`，`fix/12345-date-bug` → `12345`）
   - 如果分支名中包含数字，提取这些数字
   - 如果分支名中没有数字，则不添加数字到 commit message

4. **生成 commit message**：
   根据下面的规范生成 commit message：
   
   **格式**：`<type>: <branch-numbers><subject>`
   
   **重要**：如果分支名中包含数字，必须将数字提取出来并添加到 commit message 的开头（在 type 后面，subject 前面）
   
   **Type 类型**（必填）：
   - `feat`: 新增功能
   - `fix`: bug 修复
   - `docs`: 文档更新
   - `style`: 不影响程序逻辑的代码修改(修改空白字符，格式缩进，补全缺失的分号等，没有改变代码逻辑)
   - `refactor`: 重构代码(既没有新增功能，也没有修复 bug)
   - `perf`: 性能, 体验优化
   - `test`: 新增测试用例或是更新现有测试
   - `build`: 主要目的是修改项目构建系统(例如 glup，webpack，rollup 的配置等)的提交
   - `ci`: 主要目的是修改项目继续集成流程(例如 Travis，Jenkins，GitLab CI，Circle等)的提交
   - `chore`: 不属于以上类型的其他类型，比如构建流程, 依赖管理
   - `revert`: 回滚某个更早之前的提交
   
   **Subject**（必填）：简洁明了地说明 commit 的目的描述 长度不能超过 50 个字符  使用中文  

5. **生成示例**：
   - 如果分支名是 `feature/17245-login`，添加了新功能：`feat: 17245 添加用户登录功能`
   - 如果分支名是 `fix/12345-date-bug`，修复了 bug：`fix: 12345 修复日期显示错误的问题`
   - 如果分支名是 `refactor/99999-user-module`，重构了代码：`refactor: 99999 重构用户管理模块`
   - 如果分支名没有数字，如 `feature/login`：`feat: 添加用户登录功能`（不添加数字）

### 步骤 5: 展示 commit message 给用户确认 ⚠️ 最关键的步骤

**⚠️ 最危险的步骤 - 绝对不能跳过**：这是整个提交流程中最关键的步骤，**严禁跳过此步骤直接执行 commit 和 push**。跳过此步骤可能导致错误的代码被提交到远程仓库，造成严重后果。

**执行要求**：
1. **展示分支名和 commit message**：
   - 获取当前分支名：`git rev-parse --abbrev-ref HEAD`
   - 明确告知用户："我为您生成了以下 commit message："
   - 显示完整的 commit message
   - 说明变更的文件和内容摘要

2. **获取用户确认**（两种方式任选其一）：
   
   **方式 A：使用 AskQuestion 工具**（如果可用）：
   ```javascript
   使用 AskQuestion 工具：
   - 问题："提交分支：[分支名]，是否使用此 commit message 进行提交？"
   - 选项：["确认并提交", "修改 commit message", "取消"]
   ```
   
   **方式 B：明确询问用户**（如果 AskQuestion 工具不可用）：
   - 明确询问用户："提交分支：[分支名]，请确认是否使用此 commit message 进行提交？"
   - 提供三个选项供用户选择：
     - **选项 1**："确认并提交" - 使用此 commit message 提交并推送
     - **选项 2**："修改 commit message" - 提供新的 commit message
     - **选项 3**："取消" - 取消本次提交
   - **必须等待用户明确回复**，明确告知选择哪个选项
   - **绝对不能**在用户未明确回复的情况下执行后续步骤

3. **根据用户选择处理**：
   - **如果用户选择"确认并提交"**：
     - 继续执行步骤 6（git commit）和步骤 7（git push）
   
   - **如果用户选择"修改 commit message"**：
     - 询问用户新的 commit message
     - 使用用户提供的 message 执行步骤 6 和 7
   
   - **如果用户选择"取消"**：
     - 停止提交流程，不执行任何 git 操作

**⚠️ 绝对禁止的行为**：
- ❌ **严禁**跳过此步骤直接执行 `git commit`
- ❌ **严禁**在用户未明确确认的情况下执行 `git commit` 和 `git push`
- ❌ **严禁**假设用户同意而直接提交
- ❌ **严禁**在展示 commit message 后立即执行后续步骤

**✅ 正确做法**：
- ✅ **必须**展示 commit message 给用户
- ✅ **必须**明确询问用户是否确认
- ✅ **必须**等待用户明确回复（选择选项 1、2 或 3）
- ✅ **必须**在用户明确选择"确认并提交"后，才执行步骤 6 和 7

### 步骤 6: 执行 commit（⚠️ 必须在用户明确确认后执行）

**⚠️ 重要提醒**：此步骤**必须在步骤 5 中用户明确选择"确认并提交"后才能执行**。如果用户未确认或选择了其他选项，绝对不能执行此步骤。


### 步骤 7: 推送到远程（⚠️ 必须在用户明确确认后执行）

**⚠️ 重要提醒**：此步骤**必须在步骤 5 中用户明确选择"确认并提交"后才能执行**。如果用户未确认或选择了其他选项，绝对不能执行此步骤。

用户明确确认 commit message 后，执行：
```bash
git push
```

## 完整流程示例

```
0. 代码审核：
   - git status 和 git diff（查看待提交的代码变更）
   - 读取并应用 .cursor/skills/git-safe-commit/code-review-rules.mdc 规则
   - 根据规则进行代码审核
   - 如果发现问题：停止流程，提示用户修复
   - 如果审核通过或规则未配置：继续下一步
1. git pull
2. 检查冲突：
   - git status（检查是否有冲突文件）
   - git grep 搜索冲突标记（<<<<<<<, =======, >>>>>>>）
   - git diff 检查冲突标记（区分真正的冲突和格式问题）
   - 如果检测到真正的冲突标记：停止流程，提示用户先解决冲突
   - 如果只有格式问题（如 trailing whitespace）：继续下一步（格式问题不影响提交）
   - 如果没有冲突：继续下一步
3. git add .
4. 分析 git diff --cached 或 git diff
5. 获取当前分支名：git rev-parse --abbrev-ref HEAD
6. 从分支名中提取数字（如果有）
7. 根据变更生成 commit message（如果分支名有数字则添加到 message 中）
8. 获取当前分支名：git rev-parse --abbrev-ref HEAD
9. 展示 commit message 给用户：
   - "我为您生成了以下 commit message：[message]"
   - 说明变更的文件和内容摘要
10. 使用 AskQuestion 工具获取用户确认：
   - 问题："提交分支：[分支名]，是否使用此 commit message 进行提交？"
   - 选项：["确认并提交", "修改 commit message", "取消"]
11. 根据用户选择：
   - 如果"确认并提交"：
     - 优先使用中文 commit message 执行 commit
     - 检测退出码和 commit message 是否正确存储（验证是否有乱码）
     - 如果检测到乱码：撤销提交，转换为英文版本重试
     - 成功后执行 git push
   - 如果"修改 commit message"：询问新 message，然后执行 commit 和 push
   - 如果"取消"：停止流程
```

## 注意事项

- **严禁**在执行 git 提交流程时修改任何代码文件，只能执行 git 操作和分析，发现问题只能提示用户
- **⚠️ 最危险：严禁跳过步骤 5（用户确认步骤）直接执行 commit 和 push**。这是最严重的违规行为，可能导致错误的代码被提交到远程仓库。
- **必须**在执行 git pull 之前先进行代码审核
- **必须**读取并应用 `.cursor/skills/git-safe-commit/code-review-rules.mdc` 规则文件进行审核
- **必须**在代码审核发现问题时停止流程，提示用户先修复问题
- **必须**在 git pull 之后立即检查代码冲突
- **必须**在检测到冲突时停止流程，提示用户先解决冲突
- **必须**在 git commit 之前分析代码变更
- **⚠️ 必须**在提交前展示 commit message 给用户确认（步骤 5）
- **⚠️ 必须**等待用户明确确认后才能执行 commit 和 push
- **必须**使用 AskQuestion 工具获取用户确认（如果可用），或通过明确询问用户的方式获取确认
- **必须**提供三个选项供用户选择：["确认并提交", "修改 commit message", "取消"]
- **必须**等待用户明确选择后，才执行相应的操作
- **⚠️ 绝对禁止**在用户未明确回复的情况下执行步骤 6 和 7
- commit message 必须遵循项目规范（README.md 36-47行）
- **必须**优先使用中文 commit message，执行 commit 后必须验证 message 是否正确存储（检测乱码）；如果退出码为 0 但检测到乱码，必须撤销提交并转换为英文版本重试；如果退出码非 0 且是编码问题，则用英文重试
- 冲突检查方法：使用 `git status` 检查未合并文件，使用 `git grep` 和 `git diff` 检查冲突标记
- **重要**：区分真正的冲突标记（`<<<<<<<`、`=======`、`>>>>>>>`）和格式问题（trailing whitespace），只有真正的冲突才应该阻止提交
- 如果代码审核规则文件不存在或为空，跳过代码审核步骤
- 如果用户选择"修改 commit message"，询问用户并提供新的 message
- 如果用户选择"取消"，立即停止流程，不执行任何 git 操作
